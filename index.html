About

See where your LLM output can be trusted

When you extract structured data with an LLM at scale, some outputs are silently wrong and you do not know which. FieldTrust turns the model's own token probabilities into a per-field confidence score, then uses a small labeled sample to find the threshold where auto-accepting is statistically safe.

Lens

Inspect a single response. In structured mode every JSON field is scored and routed; in free-text mode you get a token heatmap and the alternatives the model considered. Free text is exploration only - no accept/review decision is made there.

Workspace

Load a JSONL batch to see the confidence distribution, find which schema paths are structurally weak, and fit a review policy that meets your precision target.

The guarantee

The threshold is chosen so that the one-sided Wilson lower confidence bound on auto-accept precision meets your target. "Precision was 98% on the calibration set" is an observation; "precision is at least 95% with 95% confidence" is a guarantee. The cost is coverage: raise the target and the auto-accept rate falls. That trade is the whole product.

Privacy

Everything runs client-side in this single HTML file. Nothing you paste or upload leaves your browser - there is no server to send it to.

Where the demo numbers come from

All of them are real. The samples are actual gpt-4o-mini responses, and the calibration set is 476 real extracted fields labelled against ground truth. We wrote the receipts and chose which characters to destroy, so we know what the answer was; a field that is not on the paper cannot be read, so whatever the model returns for it is a guess and the label is 0. Nothing on this page is a number we made up. Re-capture it yourself with scripts/capture-demo.js.

Limits, measured rather than assumed

On this data the score separates invented fields from read ones perfectly (AUROC 1.00), but that is an easy contrast on receipts we authored - treat it as a smoke test, not a benchmark. A real one on public receipts is the next step.

Three failure modes showed up in the real responses, and confidence catches only the first:

  • Caught: the field is missing or illegible, so the model guesses. Confidence drops and it routes to review.
  • Missed: the source is legible but wrong (OCR noise), and the model transcribes the garbage faithfully - at ~100% confidence, because it really is what the page says.
  • Missed: the source is ambiguous (03/04/05) and the model confidently applies one convention. No hesitation to detect.

So this measures whether the model was sure, which is not the same as whether it was right. It is a strong signal for the first failure mode and silent on the other two. Providers that do not expose logprobs are not supported yet (a self-consistency fallback is on the roadmap), and the guarantee holds only for data drawn like your calibration set.
