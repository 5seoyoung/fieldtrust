## 1. 제품 재정의

기존: "LLM이 문서에서 뽑아낸 JSON의 각 필드에 대해 자동 통과/검수를 판정하는 도구"
확장: **"LLM 출력의 어디를 믿을 수 있는지 시각화하고(Lens), 검수가 필요한 곳만 사람에게 보내는(Workspace) 도구"**

JSON 필드 판정은 정의에서 구현 디테일로 내려간다. 제품은 두 얼굴을 가진다:

- **Lens (진입로, 1건 탐색)**: 응답 하나를 붙여넣으면 신뢰 지도를 그려준다. 무료 체험이자 바이럴 표면.
- **Workspace (본체, 배치 운영)**: 응답 수백 건을 올리면 대시보드→검수 대기열→내보내기의 운영 루프를 제공한다. 재방문의 이유.

### 1.1 핵심 유저플로우 (플라이휠)

```
[Lens] 응답 1건 붙여넣기 → 신뢰 히트맵 "아하"
   ↓ "이걸 500건에 대해 하려면?"
[Workspace] JSONL 배치 업로드
   → 대시보드: 신뢰도 분포, 취약 필드 랭킹
   → 검수 대기열: 저신뢰 필드만, 키보드로 빠르게 (전체의 ~5%)
   → 내보내기: 수정본 JSONL + 정오 라벨 CSV
   ↓ 라벨 CSV가 캘리브레이터의 입력이 됨
[캘리브레이션] 임계값 재피팅 → 다음 배치에서 검수량 감소
```

**설계 원칙: 쓸수록 검수가 줄어든다.** 검수 행위 자체가 캘리브레이션 데이터를 생산하므로,
사용자는 도구를 쓸수록 자동통과율이 올라가는 것을 수치로 확인한다. 이것이 재방문 동기다.

## 2. Lens 사양 (v0.2)

### 2.1 입력
- 기존과 동일: OpenAI 응답 JSON(logprobs 포함) 붙여넣기 또는 샘플.
- **모드 자동 감지**: 토큰 연결 문자열에 JSON 객체가 있으면 → 구조화 모드(기존 필드 뷰).
  없으면 → **자유 텍스트 모드(신규)**. 수동 토글로 오버라이드 가능.

### 2.2 자유 텍스트 모드 (신규)
- **토큰 히트맵**: 각 토큰을 span으로 감싸고 확률에 따라 배경 알파 조정.
  기존 팔레트의 teal(확신)→amber→red(불확신) 램프 사용. 색각 접근성 위해 알파+밑줄 이중 인코딩.
- **호버 팝오버**: 해당 위치에서 모델이 고려한 대안 토큰들(top_logprobs)과 확률.
  "모델은 여기서 'March'(31%)도 고려했다" - 이 대안 노출이 교육적 핵심.
- **의심 구간 랭킹**: 문장 단위(마침표/개행 분할)로 min/mean logprob 집계,
  하위 3개 문장을 "가장 불확실한 구간"으로 상단에 리스트.
- **정직성 배너 (필수)**: 자유 텍스트 모드 진입 시 1회 노출되는 안내:
  "낮은 토큰 확률은 '틀렸다'가 아니라 '표현의 선택지가 많았다'일 수 있습니다.
  사실 오류와 가장 상관이 높은 신호는 스키마가 제약된 구조화 출력에서 나옵니다."
  → 한계를 숨기지 않는 것이 프로젝트의 신뢰성 브랜드. 구조화 모드로의 유도 역할도 겸함.

### 2.3 하지 않는 것
- 자유 텍스트에 대한 자동 통과/검수 판정은 하지 않는다 (보장 임계값은 구조화 출력 전용).
  Lens 자유 텍스트 모드는 탐색·교육 도구로 한정. 판정 기능을 붙이는 순간 과잉 주장이 된다.

## 3. Workspace 사양 (v0.2~v0.3)

### 3.1 입력 포맷 (배치 JSONL)
한 줄에 하나씩, 다음 중 하나:
```
{...OpenAI 응답 객체 전체...}
{"id": "doc-123", "response": {...OpenAI 응답 객체...}}
```
id 없으면 줄 번호로 자동 부여. 파싱 실패 줄은 건너뛰고 경고 카운트 표시.

### 3.2 대시보드 (v0.2)
- **신뢰도 분포 히스토그램**: 전체 필드의 meanLogprob 분포 (SVG 바). 현재 임계값 위치를 수직선으로 표시.
- **필드 경로별 집계 테이블**: 경로($.date 등)별 건수, 평균 신뢰도, 임계값 미달 비율.
  → "당신의 파이프라인에서 $.date가 구조적으로 약하다" 같은 인사이트. 정렬: 미달 비율 내림차순.
- **요약 스탯**: 총 문서/필드 수, 자동통과율 예상치, 검수 필요 필드 수.
- 기존 리스크-커버리지 차트와 캘리브레이터는 Workspace 안으로 이동.

### 3.3 검수 대기열 (v0.3)
- 임계값 미달 필드만, 신뢰도 오름차순(가장 위험한 것부터) 정렬.
- 한 화면에 하나: 필드 경로, 추출값, 토큰 팝오버(재사용), 원문 컨텍스트(응답에 포함된 경우).
- **키보드 우선**: a=승인(값이 맞음), e=수정(인라인 편집 후 Enter), j/k=이동, u=실행취소.
- 진행 바 + 예상 잔여. 완료 화면: "필드 2,340개 중 118개(5.0%)만 검수했고,
  나머지는 정밀도 95% 보장으로 자동 통과되었습니다."
- **세션 지속성**: IndexedDB에 진행 상태 저장. 새로고침해도 이어서 검수.
  (localStorage 금지 - 기존 결정 유지, IndexedDB 사용)

### 3.4 내보내기 (v0.3)
- **수정본 JSONL**: 원본 구조 유지, 수정된 값 반영, 필드별 `_fieldtrust: {score, decision, reviewed}` 메타 첨부(옵션).
- **라벨 CSV**: `doc_id,field_path,score,correct` - 검수에서 자동 생성 (승인=1, 수정=0).
  캘리브레이터의 CSV 입력 포맷과 호환되어 플라이휠 완성.
- **정책 JSON**: `{threshold, targetPrecision, delta, nCalib, fittedAt}` - 파이썬 패키지에서 import 가능하게.

## 4. 정보 구조(IA)와 기술 결정

### 4.1 화면 구조

랜딩 히어로는 Lens 자체 (별도 마케팅 페이지 없음 - 도구가 곧 랜딩).
상단 네비: Lens · Workspace · GitHub 링크.

### 4.2 기술 결정 (신규, DECISIONS.md에 추가할 것)
- **D-003 제안: 단일 HTML 유지 + 소스 분리.** 코드가 커지므로 src/(core.js, lens.js,
  workspace.js, ui.css)로 분리하고, node 스크립트(scripts/build.js)가 index.html로 인라인 조립.
  빌드 도구(Vite 등)는 아직 도입하지 않음 - 산출물이 단일 파일이라는 정체성 유지,
  조립 스크립트는 30줄 이내로. CI test.yml에 조립→jsdom 테스트 순서 추가.
- **D-004 제안: IndexedDB 스키마.** `sessions` 스토어: {batchHash, progress, corrections, updatedAt}.
  배치 파일의 해시로 세션 식별(같은 파일 다시 올리면 이어하기 제안).
- 클라이언트 온리 원칙 불변. 새 외부 의존성 금지(폰트 제외). 히스토그램·히트맵 모두 수제 SVG/CSS.

### 4.3 성능 가드
- 배치 상한 가이드: 1만 필드까지 무버퍼 처리 목표. 파싱은 청크 단위로 requestIdleCallback 사용,
  진행률 표시. 초과 시 경고(차단은 하지 않음).

### Sprint 2 = v0.3 (1주)

3. 내보내기 3종 (수정본 JSONL, 라벨 CSV, 정책 JSON).
4. 플라이휠 연결: 검수 완료 화면에서 "이 라벨로 임계값 재피팅" 원클릭.

수용 기준: 업로드→검수→내보내기→재피팅 전체 루프가 새로고침 포함 끊기지 않고 동작 /
라벨 CSV가 캘리브레이터에 그대로 입력됨 / jsdom 테스트에 검수 플로우 시나리오 추가.
