# 데모 데이터: 무엇이 진짜이고 무엇이 우리 것인가

> 신뢰도를 파는 도구가 지어낸 신뢰도 숫자를 보여주면 안 된다. `src/fixtures.js`의 모든 확률·추출·라벨은 실제 gpt-4o-mini 응답에서 나온다.

## 재캡처

```bash
# 키는 ~/.config/fieldtrust/openai.key 또는 OPENAI_API_KEY
node scripts/capture-demo.js && npm run build && npm test
```

비용은 gpt-4o-mini 기준 120여 호출로 몇 센트. `src/fixtures.js`는 생성물이므로 손으로 고치지 말 것.

## 진짜인 것 / 우리가 만든 것

| | 출처 |
|---|---|
| 모든 logprob, 대안 토큰, 추출 결과 | **모델** |
| 정오 라벨 (476개) | **모델 출력 vs 정답 비교** |
| 임계값, 자동통과율, LCB | **위 라벨에서 계산** |
| 영수증 텍스트, 어느 글자를 파괴할지 | **우리** |
| 산문 프롬프트 | **우리** |

핵심은 **정답을 아는 입력을 우리가 만든다**는 점이다. 영수증에 `##/##/##`라고 적혀 있으면 날짜는 종이에 없는 것이고, 모델이 무엇을 반환하든 그것은 추측이므로 라벨은 0이다. 수백 건을 손으로 라벨링하지 않고도 정직한 캘리브레이션 셋을 만들 수 있는 이유다.

**이 방식의 한계**: 우리가 만든 영수증은 하나의 템플릿에 노이즈만 다르다. "읽을 수 있음 vs 파괴됨"은 쉬운 대비다. 그래서 이 데이터의 AUROC 1.00은 **스모크 테스트지 벤치마크가 아니다.** 공개 영수증(CORD)으로 하는 진짜 벤치마크가 별도로 필요하다.

## 실제 데이터가 알려준 것 (2026-07-16)

기획서가 예상하지 못했고, 합성 데모가 가리고 있던 것들:

**1. 깨끗한 입력에는 검수할 게 없다.** 읽을 수 있는 영수증에서 모델은 전 필드 97~100%다. 제품이 가치를 갖는 지점은 **정보가 없거나 파괴됐을 때**다.

**2. 점수 분포가 극단적으로 치우친다.** 476필드 중 **347개(73%)가 정확히 logprob 0**이다. 균등폭 히스토그램은 한 막대에 93%를 몰아넣고 임계값을 그 안에 묻어버린다. 그래서 신뢰도 구간(`confidenceBuckets`)으로 바꿨다.

**3. 실제 임계값은 합성 데모의 20배 차이였다.** 합성 -0.676 vs 실제 **-0.021**. 실제 점수는 [-0.83, 0]에 산다. 합성 임계값을 실제 데이터에 쓰면 **지어낸 값이 전부 자동통과**한다. 캘리브레이션이 장식이 아니라는 증거다.

**4. `min`이 `geo`보다 잘 분리한다.** 파괴된 날짜에서 geo는 -0.123, min은 -0.711 (5.8배). 이유: `2023-10-01`은 토큰 6개인데 형식 토큰 5개가 ~100%라, 유일하게 불확실한 일(day) 토큰(57.6%)을 기하평균이 91%로 희석한다. **현재 기본 점수는 geo다.** PLAN v0.1의 "mean/min/margin 비교"가 여기서 답을 요구한다.

**5. logprob이 잡는 것과 못 잡는 것.**

| 실패 유형 | 잡히나 |
|---|---|
| 필드가 없어서 모델이 추측 | ✅ 신뢰도 하락 → 검수로 라우팅 |
| 원본이 OCR 쓰레기인데 충실히 전사 | ❌ ~100% 확신. 실제로 종이에 그렇게 쓰여 있으니까 |
| 원본이 모호한데(`03/04/05`) 한 규칙을 확신에 차서 적용 | ❌ 망설임이 없어서 감지할 신호가 없음 |

즉 이 점수는 **모델이 확신했는지**를 재지 **맞았는지**를 재지 않는다. 첫 번째 유형에는 강하고 나머지 둘에는 침묵한다. About 화면에 이대로 적어뒀다.

**6. 플라이휠에는 선택 편향이 있다.** 검수 큐에는 임계값 미달 필드만 들어간다. 그 라벨만으로 재피팅하면 절단된 표본에서 정밀도를 추정하게 되어 `feasible=false`("전부 검수")가 나온다 - 기획서 1.1이 약속한 것의 정반대다. 그래서 기존 캘리브레이션 셋에 **합치는(pool)** 방식으로 구현했다. 검수 라벨은 전부 현재 임계값 아래에 있으므로 그보다 높은 후보 임계값의 정밀도 추정을 편향시키지 않고, 경계 근처만 보강한다.

**따라서 꼬리 라벨은 임계값을 내릴 수는 있어도 올리지 못한다.** "쓸수록 검수가 줄어든다"는 자동으로 성립하지 않는다. 자동통과 영역의 정밀도를 실제로 검증하려면 **감사 표본(auto-accept된 것 중 무작위 일부도 검수)** 이 필요하다. 미해결.

## 미결정

- **기본 점수를 min으로 바꿀 것인가** (위 4번). 파이썬 정본도 함께 바꿔야 하고, CORD 벤치마크로 확정하는 게 맞다.
- **감사 표본 도입** (위 6번). 플라이휠이 약속을 지키려면 필요하다.
- 캘리브레이션 셋에 꼬리를 합치면 "auto-accept rate (calib)" 스탯이 실제 운영 자동통과율의 추정치가 아니게 된다. 배치 대시보드 수치는 여전히 유효하다.
