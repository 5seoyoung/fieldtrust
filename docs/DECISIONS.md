# 기술 결정 기록

## D-001: 웹앱은 단일 `index.html` 유지, 코어는 테스트 시 추출 (2026-07-15)

PLAN.md §13.2의 선택지 - (a) `src/core.js` 분리 + 인라인 빌드 스크립트 vs (b) 단일 파일 유지 - 중 **(b) 단일 파일 유지**를 택했다.

근거:
- v0 코어는 ~200줄이고 외부 의존성이 없다. 빌드 단계는 지금 시점에서 실패 지점만 추가하고 이득이 없다 (PLAN.md 결정 3: "기능이 늘면 Vite 전환"과 일관).
- 코어의 Node 테스트 가능성은 빌드 없이 확보했다: 코어 스크립트 블록에 `id="fieldtrust-core"` 마커를 달고, `tests/test_core.js`가 테스트 실행 시 그 블록을 추출해 평가한다. 소스는 `index.html` 하나뿐이므로 코드 중복이나 동기화 부담이 없다.
- GitHub Pages 배포가 "레포 루트 그대로"로 유지된다 (PLAN.md §9.4).

전환 조건: v0.2의 라이브 API 모드·검수 대기열이 들어오면 Vite 빌드로 전환하되 산출물은 정적 사이트 유지 (PLAN.md 결정 3).

## D-002: Python `score_fields`에 마크다운 펜스 허용 추가 (2026-07-15)

JS 포팅(`scoreFields`)은 처음부터 `{`…`}` 트리밍으로 마크다운 펜스를 허용했지만 Python 정본에는 없었다. PLAN.md 결정 2(양쪽 동일 알고리즘)와 §13.3(펜스 포함 alignment 테스트 요구)에 따라 Python 쪽에 추가했다.

구현: 우선 전체 텍스트를 그대로 파싱하고, 실패 시에만 첫 `{`~마지막 `}` 범위로 재시도. 기존에 유효하던 입력(톱레벨 배열/스칼라 포함)의 동작은 그대로 보존된다 - JS는 항상 트리밍하지만 유효한 JSON 객체 입력에서는 두 방식의 결과가 동일하다.

## D-003: 소스는 `src/`로 분리, 산출물은 여전히 단일 `index.html` (2026-07-15)

PLAN_v2.md §4.2 제안대로 채택. D-001에서 "코어가 ~200줄이라 빌드가 이득 없다"고 판단했으나, Lens 자유 텍스트 모드와 Workspace가 들어오며 코드가 3배 이상 커져 전제가 바뀌었다.

구조:
- `src/index.template.html` - HTML 셸. `INJECT:<파일>` 마커만 포함.
- `src/ui.css`, `src/core.js`, `src/lens.js`, `src/workspace.js`, `src/app.js` - 실제 소스.
- `scripts/build.js` - 마커를 파일 내용으로 치환해 `index.html` 생성. 변환·번들링 없음(20줄).

근거:
- 빌드 도구(Vite 등)는 여전히 도입하지 않는다. 조립은 문자열 치환뿐이라 의존성 0이고, "산출물은 정적 단일 파일"이라는 정체성(PLAN.md 결정 1·3)이 그대로 유지된다.
- `index.html`은 계속 커밋한다. GitHub Pages가 레포 루트를 그대로 서빙하는 배포 모델(PLAN.md §9.4)이 유지되고, 레포를 clone하지 않아도 파일 하나로 앱이 돈다.
- 커밋된 `index.html`이 `src/`와 어긋나는 것이 유일한 위험이므로, `build.js --check`를 CI(test.yml)에 넣어 드리프트를 빌드 실패로 만든다. 로컬 `npm test`는 조립 후 테스트한다.
- 테스트는 계속 조립된 `index.html`을 대상으로 한다(D-001의 `id="fieldtrust-core"` 추출 방식 유지) - 사용자가 실제로 받는 산출물을 검증하는 것이 맞다.

## D-004: 배치 세션 지속성은 IndexedDB (2026-07-15, Sprint 2에서 구현 완료)

PLAN_v2.md §4.2 제안대로 채택. `sessions` 스토어에 `{batchHash, progress, corrections, updatedAt}`를 저장하고, 배치 파일 내용의 해시(FNV-1a)로 세션을 식별한다. 같은 파일 재업로드 시 이어하기를 제안한다(자동 복원하지 않음 - 사용자가 새로 시작할 의도일 수 있다).

근거: localStorage는 5MB 제한과 동기 블로킹 때문에 수백 건 배치에 부적합. IndexedDB는 브라우저 내장이라 클라이언트 온리·의존성 0 원칙을 깨지 않는다. 데이터는 여전히 브라우저를 떠나지 않는다.

구현 시 결정 3가지:
- **저장 불가 시 무력화(degrade), 실패(throw) 아님.** 시크릿 모드·저장소 차단 환경에서 `openDb()`가 null을 반환하고 검수는 그대로 동작한다. 이어하기를 잃는 것은 감수할 수 있지만 검수가 막히는 것은 안 된다. jsdom에는 IndexedDB가 없으므로 지속성 테스트를 제외한 전 테스트가 이 폴백 경로를 자연히 검증한다.
- **연결 1개 캐시 + 쓰기 직렬화·병합.** 최초 구현은 결정마다 연결을 열고 닫았는데, Chromium에서 40회 빠르게 승인하니 **34개만 저장되고 6개가 유실**됐다(동시 open/close 경쟁). 지금은 연결을 하나 캐시하고, 쓰기를 프로미스 체인으로 직렬화하며, 대기 중 스냅샷은 최신 것으로 덮어써서 버스트가 소수의 트랜잭션으로 병합된다. 마지막 상태가 반드시 마지막에 쓰인다. 수정 후 40/40 보존 확인.
- **테스트에는 `fake-indexeddb`(devDependency).** 앱 자체의 의존성 0 원칙은 그대로다. 단, **fake-indexeddb는 위 경쟁을 재현하지 못한다** - 깨진 구현으로도 "버스트 전부 저장" 테스트가 통과한다. 그래서 결과가 아니라 **메커니즘**(연결 1회 open, 쓰기 병합)을 단언하는 테스트를 따로 뒀다. 깨진 구현에서는 open이 41회로 잡혀 실패한다. 실제 경쟁의 증거는 브라우저 실측이지 이 테스트가 아니다.

## D-006: 라벨 CSV는 4컬럼, 캘리브레이터는 2·4컬럼 모두 수용 (2026-07-15)

PLAN_v2.md §3.4는 라벨 CSV를 `doc_id,field_path,score,correct`로, §1.1은 그 CSV가 캘리브레이터 입력이 되는 플라이휠을 요구한다. 그런데 기존 캘리브레이터는 `score,correct` 2컬럼만 파싱했으므로 그대로는 플라이휠이 끊긴다(첫 컬럼이 `doc-1`이라 전 행이 스킵됨).

`parseLabelCsv`로 통합: 헤더가 있으면 이름으로 컬럼을 찾고, 없으면 **마지막 두 컬럼**을 score/correct로 읽는다. 이러면 사용자가 손으로 치는 2컬럼 형식과 검수가 뽑아낸 4컬럼 형식이 같은 코드로 처리된다. `correct`는 `1/0`, `true/false`, `yes/no`를 모두 받는다.

doc_id·field_path를 CSV에 남기는 이유: 캘리브레이션에는 안 쓰이지만, 사용자가 "어느 문서의 어느 필드가 틀렸었나"를 추적할 수 있어야 라벨셋이 재사용 가능한 자산이 된다.

## D-005: 배치 파싱 청킹(requestIdleCallback)은 도입하지 않음 (2026-07-15)

PLAN_v2.md §4.3은 "파싱은 청크 단위로 requestIdleCallback 사용, 진행률 표시"를 요구했으나, 실측 후 **구현하지 않기로** 결정했다.

측정(jsdom, M-series macOS, 업로드→대시보드 렌더 완료까지):
- 500건 / 2,000필드: **23ms**
- 2,000건 / 8,000필드: **30ms**

수용 기준이 "500건 3초 내"인데 실측이 그 100배 이상 빠르다. 목표 상한인 1만 필드에서도 프레임 하나 수준이라, 청킹·진행률 UI는 사용자가 볼 일 없는 상태 머신을 추가하는 것에 불과하다. 없는 문제를 위한 코드는 넣지 않는다.

재검토 조건: 실제 사용자 응답이 데모 배치보다 훨씬 크거나(문서당 수백 필드), 1만 필드에서 브라우저 실측이 100ms를 넘으면 그때 도입한다. 수동 체크리스트는 docs/PERF.md 참조.

## D-007: 플라이휠은 검수 큐가 아니라 균일 감사 표본으로 재피팅한다 (2026-07-18)

PLAN_v2.md 1.1의 플라이휠("쓸수록 검수가 준다")을 Sprint 2에서 "검수 라벨을 캘리브레이션 셋에 pool"로 구현했는데, 실측 결과 **통계적으로 무효**였다. 검수 큐는 정의상 임계값 미달 필드만 담기므로 절단된(censored) 표본이고, Wilson의 i.i.d. 가정을 깬다.

실제 데이터로 확인 (476필드 캘리브레이션 셋, 임계값 -0.0205):
- 검수 큐 라벨 pool: 6배치를 부어도 임계값이 **-0.0205에서 전혀 안 움직임**. "검수할수록 준다"의 정반대로, 정책이 얼어붙는다.
- 균일 2% 감사 표본 pool: 임계값이 -0.0205 → **-0.0403**으로 내려가고 커버리지가 94.1% → 94.7%로 오른다. 플라이휠이 실제로 작동.

원인: 자동통과 영역 정밀도 0.969, 검수 영역 정밀도 0.036. 검수 라벨을 넣으면 정밀도 추정이 꼬리값(0.036)에 끌려 내려가, 낮은 임계값이 계속 infeasible로 보인다.

**해법 (D-007): 검수 큐와 캘리브레이션 표본을 분리한다.**
- `reviewQueue`가 두 종류를 담는다: (1) 임계값 미달 **교정(correction)** 필드 전부, (2) 결정과 무관하게 균일 비율(`AUDIT_RATE` = 2%)로 뽑은 **감사(audit)** 표본. 감사 표본은 자동통과 영역까지 포함하므로 i.i.d.이다.
- `sampleU(key)`는 필드 해시 기반 결정론적 추출. 새로고침해도 표본이 고정되고, 마음에 들 때까지 재추첨할 수 없다.
- 감사 항목은 큐에 **끼워넣는다(interleave)**. 점수순 정렬이면 전부 끝에 몰려, 일찍 멈춘 검수자가 표본을 놓치고 패턴을 눈치챈 검수자가 도장만 찍는다. 검수자는 어느 게 감사인지 모르는 게 낫다(무편향 보장).
- `calibRows`는 `inCalib`(균일 표본)만 반환. 재피팅은 이것만 pool한다.
- `auditPrecision`은 `isAudit`(감사된 자동통과)만으로 실측 정밀도를 계산 → 완료 화면에 "자동통과 N건 중 M건 정답(P%), 목표 유지/만료" 경보.

**부수 효과**: 감사 표본은 선택 기능이 아니라 플라이휠의 **전제조건**이었다. 자동통과 영역을 실측하는 유일한 무편향 창이므로, 분포 이동 감지(보장 만료 경보)도 공짜로 따라온다. PLAN 정리의 두 미결(감사 표본 도입 / 플라이휠 유효성)은 하나의 문제였고 함께 닫혔다.

**비용**: 자동통과율 95% 기준 검수 부하 +약 1.9%. 라벨 CSV 내보내기와 원클릭 재피팅 모두 감사 표본만 사용한다. 교정 라벨은 수정본 JSONL에 반영될 뿐 캘리브레이션에 쓰이지 않는다.

**한계**: 한 배치(60문서, 2%)는 감사 라벨 ~12개라 `MIN_CALIB_ROWS`(20) 미만이라 재피팅 비활성. 여러 배치에 걸쳐 누적해야 한다. 완료 화면이 이 사실을 명시한다.
