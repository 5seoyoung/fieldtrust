# FieldTrust 프로젝트 기획안

> LLM 구조화 추출(structured extraction)의 필드 단위 신뢰도 검사 및 검수 라우팅 도구.
> 이 문서는 기획 배경과 기술 명세를 모두 담은 단일 참조 문서(single source of truth)입니다.
> 대상 독자: 이 프로젝트에 처음 합류하는 기획자, 개발자, 그리고 코딩 에이전트(Claude Code).

- 작성일: 2026-07-15
- 상태: v0 프로토타입 완성 (동작 검증 완료), 레포 이전 및 배포 준비 단계
- 소유자: 서영 (솔로 개발, SNUH 의료 AI 연구원 - UQ/신뢰성 전공)

## 1. 한 문장 정의

**"LLM이 문서에서 뽑아낸 JSON의 각 필드에 대해, '이 값은 자동 통과시켜도 정밀도 95%가 통계적으로 보장된다 / 이 값은 사람이 봐야 한다'를 판정해주는 브라우저 도구."**

엘리베이터 피치: 회사들이 영수증·계약서·리포트를 LLM으로 JSON화할 때, 1,000건 중 어느 10건이 틀렸는지 알 수 없어 전수 검수를 하거나 오류를 그냥 삼킨다. FieldTrust는 모델의 토큰 확률(logprob)을 필드 단위로 재구성하고, 사용자의 소량 라벨 데이터로 캘리브레이션하여, "여기까지는 자동, 여기부터는 검수"라는 운영 정책을 통계적 보장과 함께 제공한다. 전 과정이 브라우저 안에서 실행되어 데이터가 외부로 전송되지 않는다.

## 2. 문제 정의

### 2.1 페인포인트

LLM 구조화 추출은 프로토타입에서는 잘 동작하지만, 스케일에서는 산발적 오류(잘못된 금액, 존재하지 않는 날짜, 뒤바뀐 항목)가 발생하며 **어느 출력이 틀렸는지 모른다는 것**이 자동화의 실질적 병목이다. 현재 실무자들의 우회책은 세 가지인데 모두 나쁘다: (1) 전수 인간 검수 - 자동화의 의미가 없음, (2) 무검수 통과 - 오류가 하류 시스템(회계, DB)으로 유입, (3) raw logprob 평균이나 "모델에게 확신도 물어보기" - 캘리브레이션되지 않아 "0.7이면 믿어도 되나?"에 답할 수 없음.

### 2.2 수요 근거 (조사 시점: 2026-07)

- 프로덕션 문서 추출에서 "신뢰도 점수로 문제 출력만 플래그하여 검수 예산을 집중"하는 패턴이 업계 표준 요구로 자리잡음 (Cleanlab TLM 벤치마크 문서, Sensible 등 상용 플랫폼의 confidence signals 기능).
- LLM 문서 추출의 selective prediction은 학계에서도 미해결 문제로 지목됨 (RobustifAI@IJCAI 2026).
- logprob 단일 신호의 한계가 문서화됨: epistemic uncertainty(모델이 모른다는 것을 모르는 경우)를 놓치고, Anthropic 등 일부 프론티어 제공자는 토큰 확률 접근 자체를 제공하지 않음. → 다중 신호와 폴백 전략이 필요하다는 근거.

## 3. 타깃 사용자와 시나리오

**1차 타깃**: 문서 처리 파이프라인을 만드는 개발자 (인디 해커, 스타트업 백엔드, RPA→LLM 전환 팀).

핵심 시나리오 A (인스펙터): 개발자가 자기 파이프라인의 OpenAI 응답(logprobs 포함)을 붙여넣기 → 필드별 신뢰도가 색으로 표시된 JSON을 보고 → 어느 필드가 취약한지 즉시 파악 → 프롬프트/스키마 개선 또는 검수 정책 설계에 반영.

핵심 시나리오 B (캘리브레이터): 운영 이력에서 (필드 점수, 정오 라벨) 200~500건을 CSV로 확보 → 업로드 → 목표 정밀도(예: 95%)와 신뢰수준을 설정 → "임계값 X 이상 자동 통과 시 정밀도 95% 보장, 자동 통과율 Y%"라는 정책과 리스크-커버리지 곡선을 획득 → 파이프라인에 임계값 적용.

**2차 타깃**: 민감 데이터(의료, 금융) 때문에 SaaS에 데이터를 못 올리는 조직 - "브라우저 밖으로 데이터가 나가지 않음"이 결정적 요인.

## 4. 경쟁 지형과 차별점 (정직한 평가)

| 도구 | 성격 | 하는 것 | 안 하는 것 |
|---|---|---|---|
| structured-logprobs (arena-ai) | 오픈소스 py | OpenAI 구조화 출력의 토큰 logprob 분석, 문자→토큰 매핑 | 캘리브레이션, 임계값, UI |
| llm-confidence (VATBox) | 오픈소스 py | JSON 키-값별 logprob 집계 | 캘리브레이션, 보장, UI |
| promptrepo-score | 오픈소스 npm | 필드별 logprob→신뢰도 변환 | 캘리브레이션, 보장, UI |
| UQLM (CVS Health) | 오픈소스 py | 자유 생성 응답의 환각 탐지 스코어러 모음 | 구조화 출력 필드 단위 |
| Cleanlab TLM | **유료 API** | 벤치마크 검증된 신뢰도 점수, 다중 신호 | 오픈소스 아님, 데이터 외부 전송 |

**결론: "필드별 logprob 점수 계산"은 레드오션이다. 우리의 차별점은 그 위 레이어다.**

1. **의사결정 레이어**: 점수 계산에서 멈추지 않고, "자동 통과 정밀도 ≥ 목표치"를 통계적 하한(Wilson bound, 추후 Learn-then-Test)으로 보장하는 임계값을 산출. 기존 오픈소스 중 이것을 하는 도구는 없음.
2. **제로 인프라 + 프라이버시**: 전 기능이 정적 웹페이지 하나. 설치도 서버도 없고 데이터가 브라우저를 떠나지 않음.
3. **인터랙티브 정책 설계**: 목표 정밀도 슬라이더를 움직이면 자동통과율과 필드 판정이 실시간으로 뒤집히는 UI - "보장의 비용"을 체감하게 하는 것 자체가 제품 경험.
4. (로드맵) logprob 미제공 모델용 self-consistency 폴백 - 최신 프론티어 모델 대응.

포지셔닝 문구: 점수 라이브러리가 아니라 **"구조화 추출의 검수 정책 엔진(review policy engine)"**. 기존 오픈소스 점수 라이브러리들과는 경쟁이 아니라 그 위에 얹히는 관계로 설명한다.

## 5. 핵심 개념 용어집 (기획자용)

- **logprob**: 모델이 각 토큰을 생성할 때의 로그 확률. 0에 가까울수록 확신. 모델의 "속마음"에 해당하는 원시 신호.
- **토큰↔필드 정렬**: logprob는 토큰 단위로만 나오므로, 어느 토큰이 JSON의 어느 필드 값에 속하는지 문자 위치로 역추적하는 과정. 본 프로젝트의 기술적 핵심 1.
- **캘리브레이션**: 원시 점수를 "실제로 맞을 확률"로 변환하는 것. 점수 0.7이 정답률 70%를 의미하도록 보정.
- **selective prediction**: 모델이 확신 없는 케이스는 답하지 않고(=검수로 넘기고), 확신 있는 케이스만 답하는 프레임워크. 본 프로젝트의 이론적 기반.
- **Wilson lower bound**: 관측된 정답률의 통계적 하한. "캘리브레이션 셋에서 정밀도 98%였다"가 아니라 "95% 신뢰수준에서 정밀도가 최소 95%다"라고 말할 수 있게 해주는 장치. 본 프로젝트의 기술적 핵심 2.
- **리스크-커버리지 곡선**: 자동 통과율(커버리지)을 높일수록 정밀도가 어떻게 떨어지는지 보여주는 곡선. 정책 설계의 지도.

## 6. 현재 상태 (v0, 완성)

두 개의 산출물이 존재하며 모두 동작 검증 완료:

**(a) Python 코어 패키지** (`fieldtrust/`, pip 설치 가능)
- `json_spans.py`: 위치 추적형 JSON 파서. 모든 값의 문자 span을 JSONPath로 매핑. 중첩·배열·이스케이프·유니코드 테스트 통과.
- `alignment.py`: 토큰 오프셋↔필드 span 교차로 필드별 점수(geo/min/joint prob, top-2 margin) 산출. `strip_quotes` 옵션으로 구두점 토큰의 신호 희석 방지. OpenAI 어댑터 포함.
- `calibrate.py`: Platt 스케일링(numpy 전용 Newton법) + Wilson 하한 기반 보장 임계값 탐색.
- `examples/demo.py`: API 키 없이 도는 스모크 테스트.

**(b) 단일 파일 웹앱** (`fieldtrust.html` → 레포에서는 `index.html`)
- 코어 로직의 JS 포팅(파이썬과 동일 알고리즘) + UI. 외부 의존성은 Google Fonts뿐.
- 기능: 샘플/붙여넣기 응답 분석, 신뢰도 주석 JSON 뷰(필드별 색상+신뢰도 바), 토큰 확률 팝오버, 판정 테이블(취약 필드 우선 정렬), 캘리브레이션(합성 데모 셋 or CSV), 목표 정밀도 슬라이더 실시간 재계산, 리스크-커버리지 SVG 차트, 보장 불가 시 경고.
- jsdom 스모크 테스트 통과: 필드 판정, 중첩 경로, 임계값 피팅(데모 셋 기준 임계값 -0.676 / 자동통과 23.6% / LCB 95.0%), 슬라이더 인터랙션.
- 마크다운 펜스(```json)로 감싸진 응답도 처리.

## 7. 아키텍처와 기술 결정

**결정 1 - 클라이언트 온리.** 서버·DB·백엔드 없음. 이유: 솔로 유지비 최소화, 프라이버시 셀링포인트, GitHub Pages 무료 배포. 모든 계산(파싱, 정렬, 캘리브레이션)은 수 밀리초 수준이라 브라우저로 충분.

**결정 2 - 파이썬 코어와 JS 포팅의 이중 유지.** 파이썬은 벤치마크·연구·pip 사용자용 정본(reference), JS는 웹앱용. 알고리즘 변경 시 양쪽 동기화 필요 - 코어 로직 변경은 반드시 양쪽 테스트를 함께 통과시킬 것.

**결정 3 - 단일 HTML 파일 유지 vs 빌드 도입.** v0은 단일 파일. 라이브 API 모드, 검수 대기열 등 기능이 늘면 Vite 빌드로 전환하되, 산출물은 여전히 정적 사이트여야 함(배포 모델 불변).

**결정 4 - 언어.** 제품 UI/README는 영어(글로벌 개발자 타깃, X 오디언스와 일치). 내부 기획 문서는 한국어.

## 12. 리스크와 미결정 사항

- **logprob 접근성 축소**: 프론티어 모델들이 logprob 제공을 줄이는 추세 → v1.0의 self-consistency 폴백이 헤지. 우선순위 상향 가능.
- **CORD 벤치마크에서 AUROC 저조 가능성**: go/no-go 지점. 미달 시 다중 신호 결합으로 피벗(오히려 연구 기여 커짐).
- **이름**: "FieldTrust" 상표/패키지명 충돌 미확인. PyPI/npm/GitHub 검색 필요.

## 13. Claude Code 초기 작업 지시

이 폴더에는 `fieldtrust.zip`(파이썬 패키지)과 `fieldtrust.html`(웹앱)이 있다. 다음을 순서대로:

1. zip을 풀어 8절의 레포 구조로 재배치하라 (`fieldtrust.html` → `index.html`).
2. `index.html`의 인라인 코어 스크립트를 `src/core.js`로 추출하되, 단일 파일 배포가 유지되도록 간단한 인라인 빌드 스크립트(node) 또는 그대로 유지 중 택일하고 근거를 남겨라.
3. pytest 테스트 작성: json_spans 엣지케이스(중첩/배열/이스케이프/유니코드/빈 문자열), alignment(마크다운 펜스 포함), calibrate(Wilson 값 검증: wilson(99,100,0.05)≈0.9564).
4. node 테스트 작성: 기존 검증 시나리오(이 문서 6절 (b)) 재현.
5. GitHub Actions 워크플로 2개(test, pages) 작성.
6. 영문 README 초안: 포지셔닝은 4절의 "review policy engine" 프레이밍을 따를 것. 경쟁 도구를 폄하하지 말고 "이들 위에 얹히는 의사결정 레이어"로 서술.
7. 모든 커밋 전 테스트 통과 확인.

주의: 코어 알고리즘(파서, Wilson 탐색)은 검증된 상태이므로 동작 변경 없이 이동만 할 것. 리팩토링이 필요하면 테스트를 먼저 옮긴 후 진행.
